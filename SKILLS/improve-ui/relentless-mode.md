# Quality Loop

Use this stance for implementation work that must continue until the requested UI outcome is evidenced. It is not permission to widen scope, edit during a read-only audit, or treat taste as fact.

## Start

- Set the profile: `micro`, `focused`, or `deep`.
- State the requested bar and stop condition.
- Capture the current state before a visual, interactive, responsive, or performance change.
- Translate the visible symptom into user impact and source cause.
- Keep the user-facing path, constraints, and proof target explicit.

## Iterate

1. Fix the highest-severity in-scope issue.
2. Prefer the smallest systemic repair when instances share a cause.
3. Preserve unrelated behavior and user changes.
4. Run the narrowest relevant source/test/runtime check.
5. Inspect the same state plus one relevant edge or recovery state.
6. Repeat while evidence still shows an unresolved P0/P1 or repeated systemic P2 caused by the touched path.

Do not use repeated iteration to add unrelated redesign, dependencies, or product behavior.

## Reject False Completion

Do not stop on:

- advice-only output when the user authorized implementation and the source is editable;
- a detector-only report presented as visual judgment;
- compilation or typecheck presented as UI proof;
- a screenshot from a different route, viewport, state, theme, or content;
- state names listed without setup, action, assertion, and successful execution;
- “looks better” without an artifact or clearly limited claim;
- unresolved in-scope P0/P1 findings hidden by an aggregate score;
- a clean default state while the changed edge/recovery state still fails.

## Challenge Precisely

Use direct, evidence-backed language:

- “The main blocker is … because …”
- “The interface prioritizes X, while the task requires Y.”
- “This patch is local, but the repeated cause is …”
- “I am not calling this verified because …”

Avoid vague phrasing such as “improve hierarchy,” “reduce clutter,” or “make it modern” unless followed by exact removals, merges, promotions, and proof.

Be rigorous toward the artifact and collaborative toward the user. Match the requested tone; do not manufacture hostility.

## Stop

Stop only when one condition is true:

- evidence meets the requested bar;
- remaining issues are outside the authorized scope and recorded as risk;
- a specific blocker prevents implementation or proof;
- the user explicitly requested a quick/read-only assessment.

Use “implemented, not fully verified” or “reviewed, blocked by …” when evidence is incomplete. Never turn elapsed effort into a pass.
