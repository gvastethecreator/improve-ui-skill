# Implementation Execution Contract

Use this for every nontrivial authorized implementation. It converts the skill from advice into observable work. A coordinator may add constraints, but may not remove these gates while claiming Improve UI was applied.

## One Work Unit, One Primary Surface

A work unit owns one primary product archetype, route or tightly coupled flow. Five unrelated interfaces are five isolated work units, even when they share a benchmark, model, repository, or deadline. Do not ask one builder to improve a studio, dashboard, command center, HUD, and landing page in one pass; context switching collapses inspection into generic theming.

Shared primitives may form one work unit only when the same causal edit and proof matrix genuinely cover their consumers. Record bounded secondary regions on the context card. If they have incompatible spatial, scrolling, interaction, content, or costly-state models, split them.

## Required Builder-Owned Artifacts

Use coordinator-supplied paths when present. Otherwise keep temporary evidence in the operating-system temp directory and report exact paths; do not litter product source merely to satisfy the record.

Before the first edit, create or explicitly return `context-card.json` containing:

- work-unit id, profile, target route, primary archetype, secondary regions, user mode and primary artifact;
- frequency, pressure, input, spatial model, costly states and intended hierarchy;
- source cause, preserved contracts, constraints, state/viewport matrix and proof targets.

For a rendered change, the same builder must produce:

- `proof/before.png` at the representative state and viewport;
- `proof/after.png` at the same route, state, viewport, theme, content and auth context;
- `proof/detail.png` or named DPR 2 crops for alignment, spacing, dense rows, scrollbars, gradients and icon/vector craft;
- `finish-ledger.json` with `alignment`, `spacing-rhythm`, `overflow`, `scrollbars`, `gradients`, `icons-vector-craft`, `content-integrity`, `capture-legibility`, and `responsive-detail`.

Each ledger item records `status: passed | failed | not-applicable | blocked`, evidence paths, observation and corrective action. Alignment, spacing, overflow, content integrity, capture legibility and responsive detail are always applicable. A scrollbar may be not applicable only when no scroll region exists; when scrolling exists, default native chrome fails until a minimal theme-aware treatment is proven or a platform limitation is named. Icons and gradients may be not applicable only when absent.

Every durable JSON record must have a sibling Markdown companion generated or updated from the same facts: [context card](templates/context-card.md), [finish ledger](templates/finish-ledger.md), proof-manifest summary, and any direction/decision ledger used by the work. Markdown is the fast-ingestion and correction surface; JSON remains the machine contract. A missing or contradictory companion blocks handoff, but temporary read-only diagnostics may stay ephemeral.

## Capture, Judge, Correct

The builder owns the loop:

1. Capture before.
2. Implement the smallest systemic repair that satisfies the authorized scope.
3. Capture after and DPR 2 details.
4. Inspect the images visually at readable scale before using source to explain them.
5. Correct every applicable failed dimension and recapture the same evidence.

Coordinator screenshots may compare conditions later; they do not substitute for this loop. A screenshot file existing is only `captured`, not `passed`.

## Completion Gate

Completion is blocked when an artifact is absent, a route/control/state did not execute, an applicable finish item is `failed` or `blocked`, the before/after frames are mismatched, or the visual result still contradicts the context card. Use `implemented, not fully verified` and name the blocker. Do not record `complete`, `production-ready`, or a visual-quality win.
